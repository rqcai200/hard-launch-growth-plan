1

Foundation

Build your first prototype this week
Why

The bottleneck in product development used to be engineering capacity. Now it's the speed at which PMs can validate ideas. If you can go from insight to working prototype in an afternoon, you compress discovery cycles from weeks to days. The PMs who learn this first will set the pace for their teams.

What you'll learn
  • The three-layer mental model. Every web app has a client, server, and database. Your prototype probably only needs the client layer. Adding a database when you don't need one triples complexity and lowers output quality.
  • Tool selection by use case. Use v0 or Magic Patterns for quick UI mockups. Use Lovable or Bolt for full-stack prototypes. Save Cursor and Claude Code for when you need to work inside an existing codebase. Starting in an IDE before you've prototyped is premature.
  • Spec-first prompting. Generate a structured PRD covering user stories, implementation phases, and a design system in Claude or ChatGPT first. Drop that spec as your first prompt. Results improve dramatically versus open-ended requests like "build me an app."
  • Read code structurally, not syntactically. You don't need to write production code. You need to understand data flow, what changes when requirements shift, and how to communicate trade-offs. Mental models matter more than syntax.
  • Directive prompting. Give small, specific instructions rather than requesting whole features. Tell the tool what NOT to change. Reuse constraint prompts to prevent the tool from breaking things you already like.
Expert Tips
Actionable tip
Start with a component library, not a blank canvas. Import your company's existing UI components (via screenshots, browser extensions like HTML2React, or directly from GitHub). The starting point matters more than the prompt -- that's what separates a prototype that looks like your product from one that looks AI-generated.

2

Application

Use prototyping to make better product decisions
Why

Building a prototype is step one. Using it to change how your team makes decisions is where the real value is. PMs who prototype don't just move faster -- they communicate more clearly, align teams around shared artifacts instead of spec docs, and validate assumptions before committing engineering resources.

What you'll learn
  • The baseline-and-fork workflow. Build a reusable component library as your base, then fork for each experiment. Iterating 25+ prompts on one project creates instability. Instead, fork, test one feature, gather feedback, start fresh. Each fork takes one prompt.
  • Prototypes as communication tools. Shared prototypes align PMs, designers, and engineers faster than any doc. Designers pull confirmed requirements back into Figma. Engineers get clarity on intended behavior. But set expectations: prototype does not equal production timeline.
  • Testing AI features with real APIs. Connect OpenAI or Anthropic APIs inside your prototype to test different prompts and get customer feedback on actual AI responses -- not mocked ones. This eliminates weeks of back-and-forth with engineering on prompt iteration.
  • Codebase onboarding with AI tools. Clone your actual repo into Cursor or Claude Code, ask "tell me about this codebase," then screenshot UI issues and paste them in. PMs who can't write code can still find bugs and write specs for identified issues.
  • AI prototyping in PM interviews. Companies are testing three skills: rapid prototyping (build something in 30 min), technical system design (explain trade-offs), and root-cause analysis (debug with AI tools). Practice all three.
  • Persistent context with claude.md files. LLMs are stateless -- every new session starts fresh. Create a claude.md file with product overview, architecture decisions, API routes, and design guidelines. It acts as shared memory across sessions.
Expert Tips
Actionable tip
Run parallel agents on separate branches. Spawn 5 simultaneous prototypes using Git worktrees, each with its own branch, PR, and tests. This turns the per-request latency bottleneck into parallel throughput and prevents cascading failures.

3

Production

Ship real products and lead AI engineering conversations
Why

Most PMs plateau after learning to prototype. The gap between "impressive demo" and "production-ready" is wider than it looks, and it's where the most career leverage sits. PMs who understand production constraints, security defaults, vendor lock-in, and evaluation systems can lead AI product conversations that most PMs can't even participate in.

What you'll learn
  • Design first, then convert to full-stack. Iterate on UI in client-only mode (cheap, fast). Only add server and database after finalizing the interface. Generating all three layers together costs 3x more tokens and wastes work when you inevitably change the UI.
  • Use production-ready templates as backend reference. Templates embed security middleware, Stripe integration, unit tests, analytics, and XSS prevention. Point Claude Code at the template as reference instead of generating backend code from scratch.
  • Understand vendor lock-in. Lovable and Bolt withhold database credentials, making migration impossible without rebuilding. Replit provides credentials for portability. For serious products: prototype in Magic Patterns, build in Replit, then self-host on Render or AWS.
  • Move from gut-feel to systematic evals. Write LLM evals with three parts: eval prompt defining judge criteria, explanation requirement forcing step-by-step reasoning, output rails constraining responses to scoreable labels. Compare across prompt variants to make data-driven decisions.
  • Start agents simple. Single agent with clear tools, not multi-agent orchestration. The tool description is the most critical field -- it tells the model when to use each tool. Multi-agent systems add unnecessary complexity until the single-agent version works.
  • Security is open by default. Supabase-based tools use Row-Level Security that is completely open by default. Production apps need auth early (not retrofitted), error handling, API rate limiting, and schema validation. The data model is the most important spec element of all.
Expert Tips
Actionable tip
Optimize agents for reliability, not speed. Restrict agents to reliable data sources, set confidence score thresholds (route to humans below 0.8), and reduce creativity in system prompts. The PM's competitive advantage is identifying the right use cases -- developers can't do that alone.
